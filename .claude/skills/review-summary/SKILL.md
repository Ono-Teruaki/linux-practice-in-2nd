# review-summary

リポジトリ直下の `summary.md` を読み、Linux カーネル / Linuxのしくみ の観点で評価し、`review.md` の末尾に追記する skill。

## 手順

1. リポジトリ直下の `summary.md` を Read で全文取得する
2. 以下の観点でレビューを作る:
   - **事実の正確性**
     - 用語の意味, 因果関係, 動機 (motivation) に誤解がないか
     - 例: 「ユーザー/カーネルモード分離の目的はストレージへの同時アクセス調停」と書いてあれば、調停は副次効果で**主目的はハードウェア・他プロセスからの保護**であると訂正
     - 例: 「メモリ保護は特権命令によって実現される」と書いてあれば、**メモリ保護はページテーブルの U/S ビット (User/Supervisor bit)** によるもので命令そのものではない、と訂正
   - **理解の深さ**
     - 表面的な説明 (what) で止まっていないか、なぜ (why) が書かれているか
     - 抽象論で終わっていないか、具体例 (システムコール名, /proc のパス, sar/strace の出力 等) を示せるか
   - **抜けがち / 次に押さえると良いトピック**
     - そのトピックを理解するうえでセットで押さえると効く周辺概念を提案
     - 例: ユーザー/カーネルモードを語るなら、特権命令の例 (`hlt`, `wrmsr`, `cli` 等), `syscall` 命令の仕組み, ページテーブルの U/S ビット, `strace`/`sar` での観察方法
   - **構成・流れ**
     - 章立てに飛びがないか, 前提知識の順序が逆転していないか
3. `review.md` の **末尾に追記** する。既存内容は絶対に消さない。フォーマット:

```
## YYYY-MM-DD レビュー

### 強み
- ...

### 事実の訂正・補足
- 該当: 「<summary.md の該当箇所を引用>」
  - 訂正: <正しい説明>
  - 理由: <なぜそう言えるか。可能なら Intel SDM, カーネルソース, 書籍などのソースを添える>

### 説明の明瞭さ・深さ
- ...

### 抜けがち / 次に押さえると良いトピック
- ...

### 総評
1〜3行。学習者が次に何をやると伸びるかを添える
```

4. **`summary.md` は絶対に変更しない**。フィードバックの反映は学習者本人がやる
5. 完了したら、追加したレビューの件数 (訂正 N件 / 深さ M件 / 抜け K件) を1行で報告

## 注意事項

- 厳しすぎず甘すぎず。**学習者の理解を伸ばす**視点で書く
- 訂正は必ず「なぜ間違いか」の理由を添える (本人が再現できるように)
- 引用元 (書籍ページ, カーネルソース, Intel SDM 等) を明記できるなら明記
- summary.md がほぼ空 / 章タイトルだけの状態なら、「まだレビュー対象が無い」とだけ報告して review.md には書き込まない
- レビュー履歴は時系列で積まれていくのが目的なので、過去のレビューを参照して「前回指摘した点が改善されたか」も触れると良い
