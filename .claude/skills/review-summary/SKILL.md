# review-summary

各章ディレクトリにある `summary.md` を読み、Linux カーネル / Linuxのしくみ の観点で評価し、同じ章の `review.md` の末尾に追記する skill。

## 章ディレクトリ

このリポジトリは章ごとにディレクトリが切られている (例: `01-operating-system-overview/`, `02-process-management-1/`, ...)。各章ディレクトリに `summary.md` (学習ノート) と `review.md` (レビュー履歴) がペアで置かれている。

## 対象章の決定

引数や cwd から、レビュー対象の章ディレクトリを1つ選ぶ:

1. ユーザーが章番号や章名を指定した場合 (例: `/review-summary 03`, `/review-summary 04-memory-management`) → 一致する章ディレクトリを選ぶ
2. 引数なし & cwd がいずれかの章ディレクトリ配下 → その章を選ぶ
3. 引数なし & cwd がリポジトリルート → リポジトリ直下の章ディレクトリのうち、**summary.md が `(ここから書く)` のテンプレ状態ではないもの**を列挙し、ユーザーに「どの章をレビューしますか?」と確認

複数章まとめてレビューしたい場合は、各章について個別に手順を回す。

## 手順

1. 対象章の `summary.md` を Read で全文取得する
2. もし summary.md がテンプレ状態 (実質的に未記入) なら、「まだレビュー対象が無い」と報告して終了。review.md は変更しない
3. 以下の観点でレビューを作る:
   - **事実の正確性**
     - 用語の意味, 因果関係, 動機 (motivation) に誤解がないか
     - 例: 「ユーザー/カーネルモード分離の目的はストレージへの同時アクセス調停」と書かれていれば、調停は副次効果で**主目的はハードウェア・他プロセスからの保護**と訂正
     - 例: 「メモリ保護は特権命令によって実現される」と書かれていれば、**メモリ保護はページテーブルの U/S ビット**によるもので命令そのものではない、と訂正
   - **理解の深さ**
     - 表面的な説明 (what) で止まっていないか、なぜ (why) が書かれているか
     - 抽象論で終わっていないか、具体例 (システムコール名, /proc のパス, `sar`/`strace` の出力 等) を示せるか
   - **抜けがち / 次に押さえると良いトピック**
     - そのトピックを理解するうえでセットで押さえると効く周辺概念を提案
     - 例: ユーザー/カーネルモードを語るなら、特権命令の例 (`hlt`, `wrmsr`, `cli` 等), `syscall` 命令の仕組み, ページテーブルの U/S ビット, `strace`/`sar` での観察方法
   - **構成・流れ**
     - 章立てに飛びがないか, 前提知識の順序が逆転していないか
   - **書籍の実験コードとの対応**
     - 各章には書籍の実験コード (.py, .sh, .go, .c 等) が置かれている。summary.md が「動かして観察した結果」に触れていない場合、該当コードを使った確認方法を提案
4. 対象章の `review.md` の**末尾に追記**する。既存内容は絶対に消さない。フォーマット:

```
## YYYY-MM-DD レビュー

### 強み
- ...

### 事実の訂正・補足
- 該当: 「<summary.md の該当箇所を引用>」
  - 訂正: <正しい説明>
  - 理由: <なぜそう言えるか。可能なら Intel SDM, カーネルソース, 書籍などのソースを添える>

### 説明の明瞭さ・深さ
- ...

### 抜けがち / 次に押さえると良いトピック
- ...

### 書籍コードでの確認案
- <この章のコード (例: fork.py, sched.py)> を使って <こういう実験> をすると <この観点> が確かめられる

### 総評
1〜3行。次に何をやると伸びるかを添える
```

5. **`summary.md` は絶対に変更しない**。フィードバックの反映は学習者本人が行う
6. 完了したら「<章名> の review.md に追記しました (訂正 N件 / 深さ M件 / 抜け K件)」と1行で報告

## 注意事項

- 厳しすぎず甘すぎず。**学習者の理解を伸ばす**視点で書く
- 訂正には必ず「なぜ間違いか」の理由を添える (本人が再現できるように)
- 引用元 (書籍ページ, カーネルソース, Intel SDM 等) を明記できるなら明記
- 過去のレビューが review.md にある場合は参照し、「前回指摘した点が改善されたか」「同じ誤解が他章でも繰り返されていないか」にも触れる
- 章番号や章名のあいまい一致でユーザー指定を解釈する (`3`, `03`, `process-scheduler`, `スケジューラ` などはすべて 03-process-scheduler を指す)
