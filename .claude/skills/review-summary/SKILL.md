# review-summary

各章ディレクトリの `summary.md` を読み、**事実誤り・用語・タイポを直接 summary.md で訂正**し、**何をどう直したかの簡潔なログを review.md に追記**する skill。

ユーザーは review.md を斜め読みするだけで「自分が何を間違えていたか」「なぜそれが間違いか」が把握できる。修正後の正しい文章は summary.md にそのまま反映済み。

## 章ディレクトリ

リポジトリは章ごとにディレクトリが切られている (例: `01-operating-system-overview/`, `02-process-management-1/`)。各章に `summary.md` (学習ノート) と `review.md` (変更ログ) がペアで置かれている。

## 対象章の決定

1. ユーザーが章番号や章名を指定 (例: `/review-summary 03`, `/review-summary process-scheduler`, `/review-summary スケジューラ`) → 一致する章を選ぶ (あいまい一致を許容)
2. 引数なし & cwd がいずれかの章ディレクトリ配下 → その章
3. 引数なし & cwd がリポジトリルート → リポジトリ直下の章ディレクトリのうち、summary.md が `(ここから書く)` などのテンプレ状態でないものを列挙し、ユーザーに「どの章をレビューしますか?」と確認

## 手順

1. 対象章の `summary.md` を Read で全文取得
2. テンプレ状態 (実質的に未記入) なら、「まだレビュー対象が無い」と報告して終了
3. 以下を抽出する:
   - **A. 直接訂正する項目** (summary.md を書き換える)
     - **事実誤り**: 因果関係, 動機 (motivation), 概念の取り違え
       - 例: 「ユーザー/カーネルモード分離の目的はストレージ同時アクセスの調停」→ 主目的は「ハードウェアと他プロセスメモリからの保護」, 調停は副次効果
       - 例: 「メモリ保護は特権命令によって実現される」→ メモリ保護はページテーブルの U/S ビット
       - 例: 「静的ライブラリは実行時に組み込む」→ 「ビルド (リンク) 時に組み込む」
     - **用語の取り違え**: コマンド/ライブラリ/関数/ファイル など概念混同
       - 例: 「`cat` はライブラリ」→ 「`cat` はコマンド (実行ファイル)。libc がライブラリ」
     - **タイポ・表記揺れ**: 明らかな誤字 (例: 「DDL地獄」→「DLL地獄」, 「eche」→「echo」)
     - **論理が逆**な記述 (例: 静的のメリット欄に共有のデメリットが書かれている等) は構造を直す
   - **B. 提案のみ (summary.md には反映しない)** (review.md に書く)
     - **理解の深さ**: もう一段踏み込めるポイント (why の追加, 具体例の追加)
     - **抜けがちトピック**: セットで押さえると効く周辺概念
     - **書籍コードでの確認案**: 該当章のコードでどう観察できるか
     - **構成・流れ**: 章立ての改善
4. **A の項目は Edit ツールで summary.md を直接修正する**。原則:
   - ユーザーの文体・トーンは可能な限り保つ (主語/語尾/丁寧度を踏襲)
   - 局所的な修正で済むなら局所修正。論理が破綻していて全面書き換えが必要なときのみ大きく書き換える
   - 大きく書き換えた場合、その旨を review.md のログで明示する
   - 新規の節やトピックは追加しない (それは B の提案に回す)
   - 番号付きリストや見出しの構造は維持する
5. `review.md` の末尾に追記。既存内容は消さない。フォーマット:

```
## YYYY-MM-DD レビュー

### 修正適用 (summary.md に反映済み)
1. (該当する見出しや段落)
   - 修正前: 「<原文の該当部分を短く引用>」
   - 修正後: 「<書き換えた内容を短く引用>」
   - 理由: <なぜそう直したか。可能なら引用元 (書籍ページ, Intel SDM, カーネルソース 等) を添える>
2. ...

### 提案 (未適用 — 自分で取り込むか判断)
- 深さ: <ここをもう一段掘ると良い>
- 抜け: <セットで押さえると効くトピック>
- 実験: <この章のコード (例: pause.c, fork.py) で <こういう観察> ができる>

### 総評
1〜2行。次に何に手をつけると伸びるか
```

6. 完了後、1行で報告。形式: `<章名> の summary.md を修正 (訂正 N件) / review.md に変更ログ追加 (提案 M件)`

## 注意事項

- **summary.md の書き換えは最小侵襲**。文体は保つ。新規トピックは追加しない (提案として review.md に書くだけ)
- 訂正の理由は必ず添える。「なぜそれが正しいか」を本人が再現できるように
- 過去の review.md レビューがある場合は読む。同じ訂正を再度書かない (前回直したはずなのに戻っていれば「再発」として簡潔に書く)
- 修正の妥当性に確信が持てない場合は、訂正せず「提案」セクションに「この記述は X の意味で書いている? もし Y の意味なら ... と修正したい」と書く。**根拠が薄いまま書き換えない**
- 引用元 (書籍ページ, Intel SDM, カーネルソース等) を明記できるなら明記
- 章番号や章名のあいまい一致でユーザー指定を解釈する (`3`, `03`, `process-scheduler`, `スケジューラ` などはすべて 03-process-scheduler を指す)
