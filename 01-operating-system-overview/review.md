## 2026-05-13 レビュー

### 強み
- 「プロセス → システムコール → カーネルモード切替 → 復帰」の往復構造を端的にまとめている。1章の幹はつかめている
- 静的/共有ライブラリの **比較** を意識して書いていて、片方だけの説明で終わっていない
- 「OS が syscall のラッパー関数 (libc) を提供しているから、高級言語からは関数呼び出しで済む」という抽象化の仕組みに触れている

### 事実の訂正・補足

- **該当**:「各プロセスがCPU外部のストレージデバイスに同時にデバイス操作する際に発生しうるコンフリクトの回避や…」
  - 訂正: User/Kernel モード分離の **主目的** は「ハードウェアと他プロセスメモリからの保護」であり、「ストレージ同時アクセスの調停」は副次的な結果に過ぎない。「コンフリクトの回避」を主に挙げると、調停が一次目的のように読めてしまう。
  - 理由: 仮にプロセスが1つしか無くても、保護機構 (User/Kernel モード) は必要 — 暴走したプロセスがハードウェアを直接叩いて壊すのを防ぐため。同時アクセスの調停 (ロック, スケジューラ) はカーネル内部の機能であって、モード分離そのものの存在理由ではない。
  - 言い換え案:「プロセスがハードウェアや他プロセスのメモリを直接操作して破壊するのを防ぐため、CPU には User モードとカーネルモードがある。共有資源の調停はその結果としてカーネルが一手に担う」

- **該当**:「OSはecheやcatなどの実行コマンドをライブラリとして提供している」
  - 訂正: `echo` (タイポ: eche) や `cat` は **ライブラリではなく独立した実行ファイル (コマンド)**。`/bin/cat`, `/usr/bin/echo` などのバイナリ。ライブラリ (libc など) は C プログラムから **関数として呼び出す** もので、別物。
  - 理由: ライブラリ = 他のプログラムにリンクして使う関数の集合 (例: `printf`, `malloc`)。コマンド = シェルから単独で起動する実行ファイル。`cat` のような GNU Coreutils 群は内部で libc を使って書かれているが、それ自体はライブラリではない。
  - 言い換え案:「OS は `cat` や `echo` などの **コマンド群** と、それらが内部で利用する **ライブラリ (libc 等)** の両方を提供している。libc はシステムコールのラッパー関数を含む」

- **該当**:「静的ライブラリは実行時に実行ファイル内にライブラリ内の関数を組み込み」
  - 訂正: 「実行時」ではなく **「リンク時 (ビルド時)」**。`cc -static -o pause pause.c` を実行した時点で、`ld` (リンカ) が libc.a などの中身を実行ファイルに焼き込む。実行時には何も追加で読み込まない。
  - 理由: 静的ライブラリ (`.a`) はあらかじめオブジェクトコードのアーカイブで、リンカがビルド時に必要なシンボルを抽出して実行ファイルに統合する。実行時にロードするのは共有ライブラリ (`.so`) のみ。
  - 言い換え案:「静的ライブラリは **ビルド (リンク) 時** にライブラリ内の関数を実行ファイル内に組み込む」

- **該当**:「DDL地獄」
  - 訂正: タイポ。**DLL地獄 (DLL hell)**。さらに、これは伝統的に Windows の文脈で使われる用語で、Linux 等で同じ問題を指す場合は「共有ライブラリのバージョン不整合」「shared library hell」と呼ぶことが多い。
  - また、文脈的に「DLL 地獄」は **共有ライブラリのデメリット** であって、静的ライブラリ側に書くと意味が逆転する。現在の記述だと「静的ライブラリにこういう問題がある」と読めてしまう。
  - 言い換え案: 「**静的のデメリット**: バイナリサイズが膨らむ / ライブラリのバグ修正で全プログラムを再リンクが必要。**共有のデメリット**: 実行環境に該当 `.so` のバージョンが揃っている必要があり、不整合で動かなくなる (いわゆる DLL 地獄, 依存性地獄)」

### 説明の明瞭さ・深さ

- 「プロセス命令」という言葉が曖昧。「プロセスが行う処理のうちハードウェアやカーネル管理資源に触れるもの」と明示すると、なぜシステムコールが必要かが浮き出る
- 「ユーザーがアクセスすべきでない情報」がやや漠然。具体的には **他プロセスのメモリ** と **カーネル空間のメモリ** の2種類があり、メモリ保護はモード分離だけでなく **ページテーブルの U/S ビット (User/Supervisor bit)** で実現されている、までセットで押さえると深まる
- 「カーネルモードに移行」の **具体的なきっかけ** が書かれていない。`syscall` 命令 (x86-64) を発行した瞬間に CPU が自動的にモードを切り替える、という機械的事実に触れるとよい

### 抜けがち / 次に押さえると良いトピック

- **`ldd` コマンド**: 共有ライブラリの依存を確認する手段。`ldd ./hello` と `ldd ./hello_static` の比較は1章定番の実験
- **vdso (linux-vdso.so.1)**: `gettimeofday` などを syscall せずに済ます高速化レイヤ。`ldd` で見える「謎の .so」の正体
- **`strace`**: ユーザーモード ↔ カーネルモードの境界を直接観察するツール。1章の理解を体感に変える鍵
- **モード切替の経路は3つ**: システムコール (`syscall` 命令), 例外 (ページフォルト等), ハードウェア割り込み — システムコール以外もカーネルに入る経路がある点に触れると視野が広がる
- **特権命令の例**: `hlt`, `cli`/`sti`, `wrmsr`, `lgdt` 等 — User モードでこれらを実行するとどうなるか (#GP 例外で SIGSEGV) を知ると保護機構の意味が腹落ちする

### 書籍コードでの確認案

- `pause.c` を動的/静的でビルドして比較
  ```bash
  cc -o pause pause.c
  cc -static -o pause_static pause.c
  ls -l pause pause_static       # サイズ差を確認
  ldd pause                      # libc, vdso, ld-linux が見える
  ldd pause_static               # not a dynamic executable
  strace ./pause      &          # 起動直後の openat(libc.so.6) を観察
  strace ./pause_static &        # libc ロードが消える
  ```
  → 「ライブラリ」「動的/静的リンク」「ユーザー/カーネル境界」が一度に確認できる
- `hello.py` / `hello.go` を `strace` で覗いて、それぞれ起動時にどれだけ syscall を呼んでいるかを比較すると、ランタイムの厚みの違いが見えて面白い
- `syscall-inf-loop.py` を `strace -c` で見れば、純粋にシステムコールを叩き続けるプロセスの様子が観察できる (`%user` vs `%system` を `sar` と組み合わせると 1章 + 3章の橋渡しになる)

### 総評
幹はつかめている。次は「ライブラリ ≠ コマンド」と「静的リンクはビルド時、動的リンクは実行時」の2点を直して、`ldd` と `strace` を実際に手で叩いた結果を貼ると、文章が観察に裏付けられたものになる。1章は本書の足場なので、この章で `strace` 慣れしておくと 2〜5章が一気に読みやすくなる。

### 修正適用 (summary.md に反映済み)

1. プロセスとカーネルモード — User/Kernel モード分離の主目的
   - 修正前:「各プロセスがCPU外部のストレージデバイスに同時にデバイス操作する際に発生しうるコンフリクトの回避やユーザーがアクセスすべきでない情報を保護するために」
   - 修正後:「プロセスがハードウェアを直接操作して破壊したり、他プロセスのメモリやカーネル空間のメモリに勝手にアクセスしたりするのを防ぐために」+「ストレージなど共有資源への同時アクセスの調停は、その結果としてカーネルが一手に担う」
   - 理由: モード分離の一次目的は破壊・不正アクセスからの**保護**。同時アクセスの調停はカーネル内部の機能であって、モード分離の存在理由ではない (プロセスが1つしか無くても保護は必要)。原文だと因果が逆に読める。
   - 加えて「プロセス命令」という曖昧語を「ハードウェアやカーネル管理資源に触れる処理」に置き換え、なぜシステムコールが必要かを示した。

2. ライブラリ — コマンドとライブラリの取り違え + タイポ
   - 修正前:「OSはecheやcatなどの実行コマンドをライブラリとして提供している」
   - 修正後:「OSは `cat` や `echo` などの実行コマンドと、それらが内部で利用するライブラリ (libc など) の両方を提供している」
   - 理由: `cat`/`echo` は `/bin/`, `/usr/bin/` 配下の独立した実行ファイル (コマンド) で、ライブラリ (関数の集合) とは別物。タイポ `eche` → `echo` も併せて修正。

3. 静的ライブラリの組み込みタイミング
   - 修正前:「静的ライブラリは実行時に実行ファイル内にライブラリ内の関数を組み込み」
   - 修正後:「静的ライブラリはビルド (リンク) 時に実行ファイル内にライブラリ内の関数を組み込み」
   - 理由: 静的リンクは `ld` (リンカ) がビルド時に `.a` のオブジェクトを抽出して実行ファイルに焼き込む処理。実行時のロードが発生するのは共有ライブラリ (`.so`) のみ。

4. 「DDL地獄」の表記と帰属
   - 修正前:「DDL地獄(アップデートなどにより後方互換性が維持できなることによる依存パッケージの連鎖的なバグの発生)」
   - 修正後:「DLL地獄(共有ライブラリのアップデートなどにより後方互換性が維持できなくなり、それに依存するプログラムが連鎖的に動かなくなる問題) を回避できる」
   - 理由: `DDL` → `DLL` (Dynamic Link Library) のタイポ。さらに、原文だと DLL 地獄が「静的ライブラリで起きる問題」のように読めてしまうが、実際は逆で **共有ライブラリで起きる問題を静的リンクが回避する** という構図。「〜を回避できる」と結語して帰属を明確にした。
   - 細かい修正: 同段落の「動作させれる」→「動作させられる」(ら抜き)、「維持できな」→「維持できなく」(脱字)、共有側の「プログラム前提の問題」→「それを利用する各プログラムの問題」(意味が通る形に修正)。

### 提案 (未適用 — 自分で取り込むか判断)
- 深さ: メモリ保護は「モード分離」だけでなく **ページテーブルの U/S ビット (User/Supervisor bit)** で実現されている、までセットで押さえると、なぜ User モードのプロセスがカーネル空間を読めないのかが腹落ちする
- 深さ: カーネルモードへの遷移経路は **システムコール (`syscall` 命令) / 例外 (ページフォルト等) / ハードウェア割り込み** の3つ。syscall 以外もあると触れると視野が広がる
- 抜け: `ldd` (共有ライブラリの依存表示)、`strace` (ユーザー/カーネル境界の観察)、`linux-vdso.so.1` (syscall を省略する高速化レイヤ) はこの章で押さえておくと2〜5章が楽になる
